# [PRD] 제품 개요 및 문서 규칙 (Overview)

## 1. 제품 개요 (Product Overview)
본 프로젝트는 Java 17의 최신 기능을 활용하여 견고하고 확장 가능한 마이크로서비스 아키텍처(MSA)를 구축하는 것을 목표로 합니다. 

도메인 모델 중심의 비즈니스 로직 설계와 스펙 중심 개발을 실천합니다.

지마켓이나 쿠팡 같은 **오픈마켓(Open Market)** 모델로 도메인을 전환하는 것은 훨씬 확장성 있고 현실적인 접근입니다. 

현재의 MANAGER가 특정 기업 내부의 관리자 느낌이라면, SELLER는 플랫폼에 입점한 독립적인 사업 주체가 되기 때문입니다.

말씀하신 의도를 바탕으로 도메인 구조를 다음과 같이 재정의해볼 수 있습니다.

### 1.1. 사용자 역할 및 권한 전환
*   **구매자 (BUYER)**: 기존과 동일 (주문, 배송지 관리).
*   **판매자 (SELLER)**: 기존의 `MANAGER`를 대체. 자신의 '상점(Store)'을 운영하며 상품을 등록하고 판매 통계를 확인하는 주체.

### 1.2. 브랜드(Brand) vs 판매자(Seller)의 관계 재정립
오픈마켓 모델에서는 **'누가 파는가(Seller)'**와 **'어떤 브랜드 제품인가(Brand)'**를 분리하는 것이 핵심입니다.

*   **판매자(Seller) 중심 구조**:
    *   상품은 반드시 특정 **판매자(Seller)**에게 귀속됩니다 (1:N).
    *   브랜드관 대신 **'판매자 미니숍(Seller Store)'** 컨셉을 도입합니다. 사용자는 특정 판매자가 올린 상품들만 모아서 볼 수 있습니다.
*   **브랜드(Brand)의 위치**:
    *   브랜드는 상품의 **주요 분류 속성**으로 남습니다.
    *   한 명의 판매자가 여러 브랜드의 상품을 팔 수도 있고(편집숍), 브랜드 본사가 직접 판매자가 되어 자기 브랜드만 팔 수도 있습니다(공식 브랜드관).

### 1.3. 도메인 변경 제안 (PRD 및 ERD 관점)

| 구분         | 현재 (브랜드 중심)           | 변경 제안 (오픈마켓/판매자 중심)        |
|:-----------|:----------------------|:---------------------------|
| **핵심 엔티티** | Brand                 | **Seller (판매자/입점사)**       |
| **상품 귀속**  | Product ∈ Brand       | **Product ∈ Seller**       |
| **전시관**    | 브랜드관 (Brand Showcase) | **셀러관/미니숍 (Seller Store)** |
| **브랜드 역할** | 상품의 상위 집합 (필수)        | **상품의 메타 정보/태그 (선택적/필수)**  |
| **사용자 권한** | MANAGER (운영자)         | **SELLER (입점 판매자)**        |

### 1.4. 이렇게 바꿨을 때의 장점
1.  **현실성**: "삼성전자"라는 브랜드 제품을 "삼성공식판매점"에서도 팔고, "ABC디지털"이라는 판매자도 팔 수 있는 구조가 됩니다.
2.  **확장성**: 나중에 '판매자 정산', '판매자별 배송 정책' 등을 도입하기가 훨씬 수월해집니다.
3.  **검색 최적화**: 사용자는 '나이키' 브랜드로 검색할 수도 있고, '신뢰도 높은 A 판매자'의 상품만 골라볼 수도 있습니다.

## 2. 문서 작성 규칙 (Document Conventions)
본 문서는 요구사항의 성격에 따라 다음의 접두어를 사용하여 식별자를 관리한다.
- **UC (Use Case)**: 사용자가 시스템에서 수행하는 **능동적인 동작이나 시나리오**를 의미한다. (예: 로그인, 상품 검색 등)
- **BR (Business Rule)**: 시스템이 항상 준수해야 하는 **정적인 정책, 제약 사항, 또는 데이터의 정의**를 의미한다. (예: 브랜드 표기 원칙, 카테고리 정의 등)

```markdown
UC_ (Use Case)와 BR_ (Business Rule)은 설계 문서에서 다음과 같이 구분하여 사용하는 것이 일반적이며 더 전문적인 방식입니다.

1. UC_ (Use Case): 사용자가 시스템에서 수행하는 능동적인 동작이나 시나리오를 의미합니다.
    - 예: "로그인을 한다", "상품을 검색한다"
2. BR_ (Business Rule): 시스템이 항상 준수해야 하는 정적인 정책, 제약 사항, 또는 데이터의 정의를 의미합니다.
    - 예: "브랜드명은 한글로 표기한다", "비밀번호는 8자리 이상이어야 한다"

왜 BR_로 표기하는 것이 좋은가요?
* 명확성: 사용자의 행동(UC)과 시스템의 원칙(BR)이 섞이지 않아 문서 구조가 깔끔해집니다.
* 참조 용이성: 나중에 개발이나 테스트 시, "이 기능은 BR_BRAND_01 정책을 준수해야 함"과 같이 유스케이스에서 비즈니스 규칙을 명확하게 참조할 수 있습니다.
* 확장성: 사용자 인터페이스(UI)가 바뀌어도 비즈니스 정책(BR)은 그대로 유지되는 경우가 많아 관리 효율성이 높아집니다.
```
