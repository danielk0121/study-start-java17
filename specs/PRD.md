# [PRD] 제품 요구사항 문서 (Single Source of Truth)

## 1. 제품 개요 (Product Overview)
본 프로젝트는 Java 17의 최신 기능을 활용하여 견고하고 확장 가능한 마이크로서비스 아키텍처(MSA)를 구축하는 것을 목표로 합니다. 도메인 모델 중심의 비즈니스 로직 설계와 스펙 중심 개발을 실천합니다.

---

## 2. 문서 작성 규칙 (Document Conventions)
본 문서는 요구사항의 성격에 따라 다음의 접두어를 사용하여 식별자를 관리한다.
- **UC (Use Case)**: 사용자가 시스템에서 수행하는 **능동적인 동작이나 시나리오**를 의미한다.
  - 예: 로그인, 상품 검색 등
- **BR (Business Rule)**: 시스템이 항상 준수해야 하는 **정적인 정책, 제약 사항, 또는 데이터의 정의**를 의미한다.
  - 예: 브랜드 표기 원칙, 카테고리 정의 등

```markdown
UC_ (Use Case)와 BR_ (Business Rule)은 설계 문서에서 다음과 같이 구분하여 사용하는 것이 일반적이며 더 전문적인 방식입니다.

1. UC_ (Use Case): 사용자가 시스템에서 수행하는 능동적인 동작이나 시나리오를 의미합니다.
    - 예: "로그인을 한다", "상품을 검색한다"
2. BR_ (Business Rule): 시스템이 항상 준수해야 하는 정적인 정책, 제약 사항, 또는 데이터의 정의를 의미합니다.
    - 예: "브랜드명은 한글로 표기한다", "비밀번호는 8자리 이상이어야 한다"

왜 BR_로 표기하는 것이 좋은가요?
* 명확성: 사용자의 행동(UC)과 시스템의 원칙(BR)이 섞이지 않아 문서 구조가 깔끔해집니다.
* 참조 용이성: 나중에 개발이나 테스트 시, "이 기능은 BR_BRAND_01 정책을 준수해야 함"과 같이 유스케이스에서 비즈니스 규칙을 명확하게 참조할 수 있습니다.
* 확장성: 사용자 인터페이스(UI)가 바뀌어도 비즈니스 정책(BR)은 그대로 유지되는 경우가 많아 관리 효율성이 높아집니다.

현재 PRD.md에도 제가 임의로 BR_ 접두어를 도입하여 카테고리 정의(BR_PROD_01)와 브랜드 표기 원칙(BR_BRAND_01)을 정리해 두었습니다. 
이렇게 구분함으로써 "상품 목록 조회(UC_PROD_01)" 등의 유스케이스가 어떤 데이터 규칙을 따르는지 더 명확하게 보여줄 수 있게 되었습니다.
```

## 3. 핵심 도메인 및 유스케이스 (Core Domains & Use Cases)
*모든 비즈니스 로직과 사용자 시나리오는 여기서만 관리하며, 기술적 구현 상세는 PRD-BACKEND / PRD-FRONTEND 문서를 참고한다.*

### 3.1. 인증 및 보안 (Auth Domain)
- **UC_AUTH_01 [회원가입]**: 사용자는 이메일과 비밀번호 정보를 입력하여 회원으로 가입할 수 있다.
- **UC_AUTH_02 [로그인]**: 회원은 이메일과 비밀번호를 사용하여 시스템에 로그인할 수 있다.
- **UC_AUTH_03 [로그인 잠금 정책]**: 회원이 로그인에 5회 연속 실패할 경우, 해당 계정은 10분 동안 로그인이 제한된다.
- **UC_AUTH_04 [로그인 잠금 안내]**: 로그인 화면에서는 계정 잠금 시 잠금 해제 예정 시간을 사용자에게 안내해야 한다.
- **UC_AUTH_05 [권한 구분]**: 회원은 상품 구매자(BUYER)와 상품 관리자(MANAGER)로 구분된다.
- **UC_AUTH_06 [구매자 인가 정책]**: BUYER는 주문 생성, 자신의 배송지 관리가 가능하며 상품 정보 관리는 불가하다.
- **UC_AUTH_07 [관리자 인가 정책]**: MANAGER는 상품 정보 관리 및 전체 판매 통계 조회가 가능하며 주문 생성은 불가하다.

### 3.2. 회원 (Member Domain)
- **UC_MEM_01 [내 정보 조회]**: 회원은 자신의 이름, 이메일, 권한 정보를 확인할 수 있다.
- **UC_MEM_02 [구매자 정보 요약]**: BUYER 권한의 회원은 내 정보에서 등록된 배송지 및 주문 건수 요약 정보를 확인할 수 있다.
- **UC_MEM_03 [관리자 정보 요약]**: MANAGER 권한의 회원은 내 정보에서 담당 부서 및 담당 브랜드 정보를 확인할 수 있다.

### 3.3. 배송지 (Shipping Domain)
- **UC_SHIP_01 [배송지 등록]**: BUYER는 주소, 별명, 우편번호를 입력하여 새로운 배송지를 등록할 수 있다.
- **UC_SHIP_02 [배송지 목록 관리]**: 사용자는 등록된 여러 배송지 정보를 목록으로 조회할 수 있다.
- **UC_SHIP_03 [배송지 수정/삭제]**: 사용자는 기존에 등록된 배송지 정보를 수정하거나 삭제할 수 있다.

### 3.4. 상품 (Product Domain)

**비즈니스 규칙 (Business Rules):**
- **BR_PROD_01 [카테고리 정의]**: 시스템에서 관리하는 상품 카테고리는 다음과 같으며, 코드는 대문자 영문을 사용한다.
    - 전자기기 (ELECTRONICS), 의류 (CLOTHING), 식품 (FOOD), 도서 (BOOKS), 기타 (ETC)

**유스케이스 (Use Case):**
- **UC_PROD_01 [상품 목록 조회]**: 사용자는 카테고리별, 브랜드별로 상품 목록을 조회할 수 있다.
- **UC_PROD_02 [상품 목록 정보 노출]**: 상품 목록에는 대표 썸네일 1개, 상품 ID, 브랜드 썸네일, 브랜드명, 상품명, 재고 수량, 가격, 판매 개수가 노출된다.
- **UC_PROD_03 [상품 검색]**: 사용자는 상품명, 브랜드명, 카테고리 키워드로 상품을 검색할 수 있다.
- **UC_PROD_04 [상품 상세 조회]**: 사용자는 특정 상품의 상세 페이지로 이동하여 상세 정보를 확인할 수 있다.
- **UC_PROD_05 [상품 상세 정보 노출]**: 상품 상세에는 상품 ID, 브랜드 썸네일, 브랜드명, 상품명, 가격, 재고, 카테고리, 썸네일 이미지(3장), 상세 이미지(3장), 판매 개수가 모두 노출된다.
- **UC_PROD_06 [재고 제어]**: 상품 주문 시 해당 상품의 재고가 실시간으로 감소하며, 재고 부족 시 주문이 차단된다.

### 3.5. 브랜드 (Brand Domain)

**비즈니스 규칙 (Business Rules):**
- **BR_BRAND_01 [브랜드 표기 원칙]**: 브랜드명은 한국어 공식 명칭(한글)을 최우선으로 사용하며, 한국 브랜드(예: CJ)나 관용적으로 영문을 사용하는 경우에만 예외를 둔다.
    - 한글 표기 예시: 애플, 삼성, 소니, 나이키, 로지텍, 유니클로, 자라, 스타벅스, 피어슨, 오라일리, 무지, 레고, 코카콜라

**유스케이스 (Use Case):**
- **UC_BRAND_01 [브랜드 관리]**: 모든 상품은 특정 브랜드에 소속되어야 한다 (1:N 관계).
- **UC_BRAND_02 [브랜드 목록 조회]**: 사용자는 브랜드 이름과 브랜드 썸네일 이미지가 포함된 브랜드 목록을 조회할 수 있다.
- **UC_BRAND_03 [브랜드별 상품 정보 노출]**: 브랜드관에서는 각 브랜드별로 등록된 상품 정보(ID, 상품명, 가격, 썸네일)와 총 상품 개수를 동시에 확인할 수 있다.
- **UC_BRAND_04 [브랜드관 검색]**: 사용자는 브랜드명이나 상품명 키워드로 브랜드관 내 정보를 검색할 수 있다.

### 3.6. 장바구니 (Cart Domain)
- **UC_CART_01 [장바구니 담기]**: BUYER는 상품 상세 페이지에서 수량을 지정하여 상품을 장바구니에 담을 수 있다.
- **UC_CART_02 [장바구니 목록 조회]**: 사용자는 장바구니에 담긴 상품들의 ID, 썸네일, 상품명, 가격, 수량, 합계 금액을 확인할 수 있다.
- **UC_CART_03 [장바구니 수량 변경]**: 사용자는 장바구니 목록에서 각 상품의 수량을 조절할 수 있다.
- **UC_CART_04 [장바구니 항목 삭제]**: 사용자는 장바구니에서 특정 상품을 제거할 수 있다.
- **UC_CART_05 [장바구니 전환]**: 주문이 완료되면 해당 상품들은 장바구니 목록에서 자동으로 제거된다.

### 3.7. 주문 (Order Domain)
- **UC_ORD_01 [주문 생성]**: BUYER는 하나 이상의 상품을 선택하고 관리 중인 배송지를 지정하여 주문을 생성한다.
- **UC_ORD_02 [주문번호 생성 규칙]**: 주문 시 'yyMMddHHmmssNNN'(연월일시분초+랜덤3자리) 형식의 고유한 주문번호가 발급된다.
- **UC_ORD_03 [주문 상태 관리]**: 주문은 PENDING(대기), CONFIRMED(확정), CANCELLED(취소) 등의 상태 단계를 가진다.
- **UC_ORD_04 [주문 목록 조회]**: 사용자는 자신의 주문 내역을 목록으로 조회할 수 있다.
- **UC_ORD_05 [주문 목록 정보 노출]**: 주문 목록에는 주문번호, 배송지 정보, 상태, 주문일시, 주문 상품 정보(ID, 상품명, 수량, 썸네일)가 노출된다.
- **UC_ORD_06 [주문 내역 검색]**: 사용자는 주문 목록에서 상품명 키워드를 입력하여 관련 주문을 검색할 수 있다.
- **UC_ORD_07 [주문 취소 권한]**: 사용자는 PENDING(대기) 상태의 주문에 한해 직접 취소를 요청할 수 있다.
- **UC_ORD_08 [주문 취소 버튼 정책]**: 주문 상세에서 취소 버튼은 항상 노출되되, 취소 불가능한 상태에서는 비활성화 처리된다.
- **UC_ORD_09 [주문 상세 조회]**: 사용자는 특정 주문의 상세 페이지에서 주문번호, 배송지, 상세 상품 목록, 배송비, 결제 총액을 확인할 수 있다.
- **UC_ORD_10 [주문 상태 타임라인]**: 주문 상세에서는 주문 상태가 변화한 내역을 시간순으로 나열한 타임라인을 확인할 수 있다.
- **UC_ORD_11 [배송비 정책]**: 각 주문에는 배송비가 포함되며, 배송비는 주문 생성 시 2,000원 이상 3,000원 이하 범위에서 결정된다. 결정된 배송비는 주문과 함께 저장되고 이후 변경되지 않는다.
- **UC_ORD_12 [배송비 노출]**: 장바구니 결제 예정 금액 및 주문 상세의 결제 금액 영역에는 상품 금액 합계와 배송비를 구분하여 표시하고, 총 결제 금액(상품 합계 + 배송비)을 함께 노출한다.

### 3.8. 판매 통계 (Sales Statistics Domain)
- **UC_STAT_01 [판매 내역 조회]**: MANAGER는 전체 상품의 판매 내역을 목록(표)으로 확인할 수 있다.
- **UC_STAT_02 [판매 내역 정보 노출]**: 판매 내역 목록에는 주문번호, 상품 ID, 상품명, 브랜드명, 수량, 판매금액, 판매일시가 노출된다.
- **UC_STAT_03 [판매 내역 상세 검색]**: MANAGER는 날짜 범위, 상품명 키워드, 판매 금액 범위를 조건으로 판매 내역을 상세 검색할 수 있다.
