# [PRD] Java 17 기반 마이크로서비스 커머스 플랫폼 (study-start-java17)

## 0. 표준 작업 프로세스 (Development Workflow)
요구사항 변경 시 다음 순서를 엄수한다. **스펙이 코드보다 우선**하며, **도메인 모델이 설계의 중심**이 되어야 한다.

1. **PRD 수정** ([`specs/PRD.md`](./PRD.md))
   - 비즈니스 유스케이스와 기술 명세를 업데이트한다.
   - 프로젝트의 방향과 '무엇을(What)' 할지 정의하는 최상위 단계이다.
2. **DB 스펙 수정** ([`specs/erd.dbml`](./erd.dbml))
   - 도메인 모델 및 DB 스키마를 설계한다. (DDD)
   - 데이터 구조와 관계가 명확해야 API와 이벤트의 기초를 다질 수 있다.
3. **API 스펙 수정** ([`specs/openapi/`](./openapi/))
   - 서비스별 YAML 파일을 수정하여 인터페이스를 정의한다. (SDD)
   - 확정된 도메인 모델을 외부에 노출하기 위한 계약(Contract) 단계이다.
4. **이벤트 명세 수정** ([`specs/events.toml`](./events.toml))
   - 서비스 간 비동기 메시지 구조를 정의한다. (EDD)
   - 시스템 간의 데이터 흐름과 서비스 간 결합도를 최종 확정한다.
5. **구현 및 검증**
   - 설계 완료 후 실제 Java 코드를 구현하고 테스트를 수행한다.
   - 설계도를 코드로 옮기는 과정이며, 무분별한 코드 수정을 방지한다.

---

## 1. 제품 개요 (Product Overview)

본 프로젝트는 Java 17의 최신 기능을 활용하여 견고하고 확장 가능한 마이크로서비스 아키텍처(MSA)를 구축하는 것을 목표로 합니다. 

특히 **Lombok 의존성을 최소화**하고 Java **Record**를 적극 활용합니다.

**도메인 모델 중심의 비즈니스 로직 설계**와 **OpenAPI/DBML 기반의 스펙 중심 개발**을 실천합니다.

---

## 2. 개발 방법론 (Development Methodologies)

### 2.1. 스펙 주도 개발 (SDD - Spec Driven Development)
- **기준 문서**: `specs/openapi/*.yaml` (서비스별 독립 관리)
- **원칙**:
    - Controller 코드를 먼저 작성하지 않는다.
    - 기능 변경 시 해당 서비스의 `openapi/*.yaml` 스펙을 우선 수정한다.
    - 구현된 코드는 정의된 API 스펙을 100% 준수해야 한다.

### 2.2. 도메인 주도 개발 (DDD - Domain Driven Development)
- **기준 문서**: `specs/erd.dbml`
- **원칙**:
    - DB 설계 및 Entity 작성 시 `erd.dbml`을 최우선 참고 및 수정한다.
    - `erd.dbml` 명세에 맞춰 기존 Repository나 Entity를 과감히 삭제하거나 변경할 수 있다.
    - 비즈니스 로직은 최대한 도메인 모델(Entity, record) 내부에 응집시킨다.

### 2.3. 이벤트 주도 개발 (EDD - Event Driven Development)
- **기준 문서**: `specs/events.toml`
- **메커니즘**: Redis Streams를 메시지 브로커로 활용하여 서비스 간 비동기 이벤트를 처리한다.
- **명세 형식**: 이벤트를 정의할 때는 [이벤트명 / 스트림 키 / 페이로드 구조(record) / 발행자 / 구독자] 정보를 명시한다.

## 3. 핵심 도메인 및 요구사항 (Core Domains & Requirements)
*이 섹션은 구체적인 기술 명세를 배제하고 순수 비즈니스 유스케이스만 나열합니다.*

### 3.1. 인증 및 보안 (Auth Domain)
- **회원가입**: 사용자는 이메일과 비밀번호 정보를 입력하여 회원으로 가입할 수 있다.
- **로그인**: 회원은 이메일과 비밀번호를 사용하여 시스템에 로그인할 수 있다.
- **권한 구분**: 회원은 **상품 구매자(BUYER)**와 **상품 관리자(MANAGER)**로 구분된다.
- **인가**: 
    - 각 서비스는 요청자의 권한을 확인하여 허용된 기능만 수행할 수 있도록 제한한다.
    - **상품 구매자**: 주문 생성, 자신의 배송지 관리 가능. 상품 정보 관리 불가.
    - **상품 관리자**: 상품 정보 생성/수정/삭제 가능. 주문 생성 불가.

### 3.2. 회원 (Member Domain)
- **프로필**: 회원은 이메일, 이름, 권한 정보를 가진다.
- **내 정보 조회**: 회원은 자신의 프로필 정보를 확인할 수 있다.
- **목록 조회**: 시스템 관리자는 전체 회원 목록을 페이지 단위로 조회할 수 있다.
- **배송지 관리**: 회원은 주소, 별명, 우편번호를 포함하는 배송지 정보를 생성, 수정, 삭제할 수 있으며 목록으로 관리한다.

### 3.3. 상품 (Product Domain)
- **상품 정보**: 상품은 이름, 가격, 재고, 카테고리, **브랜드** 정보를 가진다.
- **브랜드 관리**: 하나의 브랜드는 여러 상품을 포함할 수 있으며(1:N), 상품은 반드시 하나의 브랜드에 소속된다.
- **재고 제어**: 상품 주문 시 해당 상품의 재고가 감소하며, 재고가 부족하면 주문할 수 없다.

### 3.4. 장바구니 (Cart Domain)
- **담기**: 구매자는 상품 목록이나 상세 페이지에서 상품을 장바구니에 담을 수 있다.
- **조회**: 구매자는 자신의 장바구니에 담긴 상품 목록과 수량, 합계 금액을 확인할 수 있다.
- **수정**: 장바구니 내 상품의 수량을 변경하거나 특정 상품을 삭제할 수 있다.
- **비우기**: 주문이 완료되거나 사용자가 요청할 경우 장바구니를 비운다.

### 3.5. 주문 (Order Domain)
- **주문 생성**: 상품 구매자 권한을 가진 회원만 하나 이상의 상품을 선택(또는 장바구니에서 전환)하고 관리 중인 배송지를 지정하여 주문을 생성할 수 있다.
- **주문 상태**: 주문은 생성, 완료, 취소 등의 생명주기를 가진다.
- **이벤트 전파**: 주문 상태가 변경되면 관련된 다른 도메인(통계 등)에 이 사실을 알린다.

## 4. 기술 구현 명세 (Technical Specifications)
*이 섹션은 도메인 요구사항을 실현하기 위한 구체적인 소프트웨어 기술 명세를 기록합니다.*

### 4.1. 보안 및 인증 구현
- **비밀번호 저장**: 회원가입 시 비밀번호는 **BCrypt**로 해싱하여 저장한다.
- **관리자 가입 보안**: 상품 관리자(MANAGER) 권한으로 가입 시, 사전에 정의된 **관리자 인증 코드**를 검증하여 일치할 경우에만 권한을 부여한다.
- **로그인 메커니즘**: 로그인 성공 시 **Access Token**과 **Refresh Token**을 발급하여 인증을 유지한다.
- **JWT 인가**: 모든 내부/외부 요청의 인증 정보는 JWT를 기반으로 검증하며, 토큰 내 권한 정보를 기반으로 기능 접근을 제어한다.

### 4.2. 데이터 및 인터페이스 구현
- **배송지 정보 복사**: 주문 생성 시점의 배송지(주소, 우편번호) 정보를 주문 데이터에 복사하여 보관한다. (배송지 정보 변경 시에도 과거 주문 정보 유지 목적)
- **통계 쿼리**: QueryDSL을 활용하여 복합적인 데이터 통계 기능을 구현한다.
- **분산 추적**: Sleuth와 Zipkin을 연동하여 서비스 간 호출 흐름을 가시화한다.
- **페이지네이션**: Spring Data의 `Pageable`과 `Page<T>`를 표준으로 사용한다.

## 5. 비기능 요구사항 (Non-Functional Requirements)
- **관측 가능성(Observability)**: 모든 요청은 `traceId`를 로그에 포함하며, Zipkin을 통해 시각화한다.
- **성능**: 쓰기 작업과 읽기 작업을 `@Transactional(readOnly=true)`로 명확히 분리하여 DB 부하를 최적화한다.
- **표준 응답**: 성공 시 데이터 객체, 실패 시 `ErrorResponse` record 형식을 전역적으로 사용한다.

## 6. E2E 테스트 가이드 (E2E Testing Guide)
- **테스트 기준**: 본 문서의 **핵심 도메인 및 요구사항(섹션 3)**에 정의된 유스케이스를 기준으로 전체 비즈니스 흐름을 검증한다.
- **테스트 방법**: **Locust**와 같은 자동화 테스트 도구를 사용하여 시나리오 기반의 E2E 테스트 및 부하 테스트를 수행한다.
- **주요 시나리오**:
    1. 회원가입(BUYER) -> 로그인 -> 배송지 등록 -> 상품 조회 -> 주문 생성 및 완료 검증.
    2. 회원가입(MANAGER) -> 로그인 -> 상품 등록/수정/삭제 권한 확인 및 주문 시도 시 거부 확인.
    3. 주문 완료 후 발생하는 이벤트가 각 서비스(Member, Stats 등)에 정상적으로 전파되는지 확인.

## 7. 도메인 모델링 가이드 (DDD Context)
- **Entity**: 식별자(ID)를 가지며 생명주기가 있는 객체 (Member, Product, Order, ShippingAddress).
- **Value Object (VO)**: 속성값으로만 정의되는 불변 객체 (Address, Money 등). Java `record` 사용.
- **Aggregate**: 데이터 변경의 단위. `Order`와 `OrderItem`은 하나의 애그리거트로 묶이며, 외부에서는 `Order`를 통해서만 접근한다.
