# 연구조사: Rate Limiting / Throttling

> API Gateway 레벨에서 클라이언트별 요청 수를 제한하는 방법을 조사한다.
> CI/CD는 의도적으로 제외했다.

---

## 1. 무엇인가

API Gateway나 Nginx 앞단에서 클라이언트가 일정 시간 내에 보낼 수 있는 요청 수를 제한하는 기능이다.

```
클라이언트 A → 1초에 100번 요청
              ↓
        Rate Limiter
              ↓
        100번까지만 통과, 초과분은 429 (Too Many Requests) 응답
```

---

## 2. 왜 필요한가

- 특정 클라이언트가 서버 자원을 독점하는 걸 막음
- DDoS 완화
- 유료 API의 경우 플랜별 요청 수 제한 (무료 100회/일, 유료 10000회/일)

---

## 3. 어디서 처리하나

| 위치 | 도구 | 특징 |
|---|---|---|
| API Gateway 앞단 | Nginx, AWS API Gateway | 애플리케이션 코드 밖에서 처리 |
| 애플리케이션 안 | Spring + Bucket4j, Resilience4j RateLimiter | 세밀한 제어 가능 (사용자별, API별) |

---

## 4. 이 프로젝트와의 관계

이 프로젝트 수준에서는 Nginx 설정 한 줄로 끝나는 문제라 별도로 깊게 다룰 필요는 없다.
실제로 클라이언트별 세밀한 제한이 필요해지는 시점(유료 플랜 구분, 악성 트래픽 대응 등)이 되면 그때 Bucket4j 같은 라이브러리를 들여다보는 게 현실적이다.
